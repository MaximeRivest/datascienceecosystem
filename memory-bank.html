MVT 1.6: Greedy Shared Memory Probe

This page estimates how much shared memory your browser tab can hold safely by allocating multiple SharedArrayBuffer shards and touching one byte per memory page. It is designed to answer: “How much RAM can our browser-native data science runtime use?”

Checking SharedArrayBuffer requirements...
This is a stress test. Leave headroom for the OS and browser. Do not aim for literal 100% of system RAM.
Probe setup

Recommended: start with 256 MiB shards and a 6 GiB target. If stable, try larger shards later. Open Chrome Task Manager while this runs.

Interpretation

Total allocated is the raw size of the retained shared-memory bank. A conservative starting point for Recommended usable is total allocated minus the larger of 2 shards or 15%, because you still need room for the browser, the page itself, workers, compiled Wasm code, GPU buffers, and the rest of the OS.

If you later build the runtime around Wasm32, remember: a single Wasm memory is usually limited to <= 4 GiB. To use more than that overall, use a bank of shared shards.
